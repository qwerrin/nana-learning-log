## 1. この資料は誰のために書かれているか

この資料は、**「TypeScript + React + Tailwind CSS でWebアプリを作れるようになりたい人」**のために書かれています。

### 想定する読者像

- ✅ HTML/CSS の基礎を学んだ(Flexbox・Grid・レスポンシブ・ダークモード対応が書ける程度)
- ✅ プログラミング経験は浅め(Pythonを少し触ったことがある程度でOK)
- ✅ JavaScript は未経験または学習中でOK(この資料で一緒に学びます)
- ✅ 1人で学習を進めたい(この資料だけで完結するように設計されています)
- ❌ 既にTypeScriptを実務で使っている上級者向けではない
- ❌ HTML/CSSが全く初めての人は、先にHTML/CSS超速習プラン(Day 1〜7)から

### この資料の特徴

**JavaScriptとTypeScriptを最初から一緒に学びます。**

多くの入門資料は「まずJavaScriptを学び、次にTypeScriptを学ぶ」という2段階構成です。しかしこの資料では、**最初からTypeScriptで書くことで、型を「後付けするもの」ではなく「最初から一緒に育てるもの」として身につけます**。

実際の現場でもTypeScriptだけで書くことがほとんどです。「JSで覚えてからTSに移行する」という遠回りをしない方が、最終的に早く到達できます。

### この資料で達成できる状態

学習を終えると、次のことができるようになります。

- TypeScript プロジェクトをゼロから立ち上げられる
- `any` を使わずに型を書ける
- 関数・クラス・オブジェクトに正しく型をつけられる
- ジェネリクスやユーティリティ型を読み書きできる
- React コンポーネントに Props・Hooks の型をつけられる
- Tailwind CSS で素早くUIを作れる
- Express で簡単な API サーバーを書ける
- TypeScriptのエラーメッセージを読んで自力で解決できる

## 3. 効果的な進め方

### 3-1. 順番に進めるのが基本

第1章から第19章まで**順番に**進めることを推奨します。後半の章は前半の内容を前提にしているからです。

ただし、**完璧主義は禁物**。1回ですべて理解する必要はありません。「だいたい分かった」で次に進み、必要になったら戻る、というスタイルで十分です。

### 3-2. 1日のサイクル(目安)

```
1. 章を読む(20〜40分)               ← 概念をインプット
2. コード例を写経して動かす(20〜40分) ← 手を動かす
3. 章末の演習を解く(20〜40分)        ← 自分の力で書く
4. (任意)学んだことをメモする(10分)   ← 言語化で定着
```

**「読むだけ」で進めず、必ず手を動かす**のがコツです。プログラミングはスポーツや楽器に近く、頭で分かっても指が動かないと意味がありません。

### 3-3. コード例の扱い

- **写経推奨**: コピペではなく、自分の手で打ってください。タイポからエラーメッセージを読む練習にもなります
- **アレンジ歓迎**: コード例の値や名前を変えて遊んでみてください
- **エラーを恐れない**: TypeScriptは「エラーを出すための言語」です。赤線を見たら「教えてくれてありがとう」と思いましょう

### 3-4. 型チェックと実行の使い分け

この資料では以下のコマンドを使い分けます。

```bash
npm run dev       # ファイル変更を監視して自動実行(開発中はこれ)
npm start         # 1回だけ実行
npm run typecheck # 型チェックだけ実行(実行はしない)
```

**重要**: `npm start` や `npm run dev` は型チェックをしません。型エラーは **VS Codeの赤線** または `npm run typecheck` で確認します。

## 5. 詰まったときの対処法

学習中に「分からない」「動かない」状況は**必ず**発生します。順番に試してください。

### ステップ1: VS Codeのエラーを読む

赤線にカーソルを合わせると、エラーの説明が出ます。英語でも諦めずに読みましょう。TypeScriptのエラーメッセージは比較的丁寧で、解決のヒントが書いてあります。よくあるパターンは**第18章**にまとめています。

### ステップ2: 該当章を読み直す

「分かったつもりだったけど…」となったら、該当章をもう一度読んでください。2回目の方が理解が深まるのが普通です。

### ステップ3: 公式ドキュメントを見る

- **サバイバルTypeScript** (https://typescriptbook.jp/) — 日本語の無料オンライン書籍
- **TypeScript公式ハンドブック** (https://www.typescriptlang.org/docs/) — 英語ですが情報量が最多

### ステップ4: エラーメッセージで検索

`TS2322 Type 'string' is not assignable to type 'number'` のようなエラーコードをそのまま検索すると、Qiita・Zenn・Stack Overflowで解決策が見つかります。

### ステップ5: 30分悩んだら一旦離れる

寝かせると解決することがよくあります。シャワーや散歩のあとに「あ、そういうことか」と気づくのは脳科学的にも理にかなっています。

### ステップ6: AIに聞く

エラーメッセージをそのまま貼り付けて「これはどういう意味ですか?」と聞くと的確に教えてくれます。ただし**答えをコピペするだけにならず、なぜそうなるかを自分で説明できるレベルまで理解する**のが大事です。

## 7. 心構え

### 完璧を目指さない

プログラミング学習は山登りのようなものです。8合目あたりで「自分はまだ何も分かっていない」と感じることがあります。これは成長の証で、ベテランほど「自分は知らないことだらけ」と感じています。

「今の自分にできるところまで」で十分。TypeScriptの完全マスターを目指さず、まずは**「動くものが作れる」**を目指しましょう。

### 楽しむ

エラーが出ても、思った通りに動かなくても、**試行錯誤を楽しむ**のがプログラミングです。「なんで動かないんだ!」と苛立つときもあれば、「動いた!」と感動するときもあります。後者を増やすために、前者を経験するのです。

### アウトプットする

学んだことは、**人に説明したり、ブログに書いたり、SNSで発信する**と定着します。完璧でなくても「こんなことを学びました」と書くだけで効果があります。
