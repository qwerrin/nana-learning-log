### 2026/05/22 — rem と px の使い分け

**状況**: 教材で rem・em・px が出てきたが、いつどれを使うか判断できなかった。

**原因**: 単位ごとの仕組みは読めば分かるが、「実用上どう選ぶか」の指針が
教材だけでは見えなかった。

**解決**:
- `rem` = ルート要素（`<html>`）の font-size 基準
- 1rem = 16px は固定ではなく、ブラウザのデフォルトに依存する（ユーザー設定で変わる）
- **文字サイズや余白には rem**、**ボーダー・小さな角丸・影には px** を使うのが定石
- `em` は当面は使わなくてよい（その要素のフォントサイズ基準で文脈依存になりやすい）

**教訓**:
- 「ユーザーが文字を大きくしたとき、比例して大きくなってほしいか?」で判断する
- 余白は比例してほしい → rem、1px の線は比例してほしくない → px

### 2026/05/22 — CSS 変数の命名で何を基準にするか

**状況**: CSS 変数を `--blue` にすべきか `--primary` にすべきか、
そもそも何を変数化するか判断できなかった。

**原因**: 教材の例は `--color-primary` などきれいに付いていたが、
なぜそう付けるかの理由が自分の中で言語化できていなかった。

**解決**:
- **見た目（色そのもの）ではなく役割で命名する**
  - `--blue` ではなく `--color-primary`
  - 後で色を変えたときに名前と値がズレない
- 何を変数化するかは「**2 回以上使う or 一括変更したくなりそう**」を基準に
- 1 回しか使わない値や、変更想定のない値はベタ書きで OK

**教訓**:
- 命名は「これは何色か」ではなく「これは何のための色か」で考える
- 「3 ヶ月後の自分が変えたくなりそうな値」だけ変数化する

---

## 定着させたいこと

- **rem を主役、px を脇役、em は気にしない**（Day 4 段階のルール）
- CSS 変数は **`:root` に書く**のが慣習
- 命名は **役割ベース**（`--color-primary`、`--space-md`、`--radius-md`）
- `var(--変数名)` で参照する

---

## よく使うパターン

```css
/* デザイントークンの定型 */
:root {
  /* カラー（役割で命名） */
  --color-primary: #3b82f6;
  --color-text: #1f2937;
  --color-bg: #ffffff;
  --color-border: #e5e7eb;

  /* スペーシング（rem 基準） */
  --space-1: 0.25rem;
  --space-2: 0.5rem;
  --space-4: 1rem;
  --space-8: 2rem;

  /* その他 */
  --radius-md: 0.5rem;
}

/* 使う側 */
.card {
  padding: var(--space-4);
  border: 1px solid var(--color-border);
  border-radius: var(--radius-md);
  color: var(--color-text);
}
```

### 2026/05/22 — Flexbox の入れ子

**状況**: プロフィールカードで `.profile-card` と `.avatar` の両方に `display: flex` が出てきて、何が何だかわからなくなった。

**原因**: Flex は「直接の子しか並べられない」ので、別々の階層で並べたいときはその階層ごとに `display: flex` が必要になる。今回は「アバターと情報を並べる」のと「アバターの中の文字を中央に置く」という2つの並べたい場所があったので、Flex が2階層にネストしていた。

**解決**: それぞれの Flex の役目を分けて理解した。
- `.profile-card` の Flex: アバターと情報を横に並べる
- `.avatar` の Flex: アバターの中の文字を中央に置く

**教訓**: Flex は「並べたいモノたちの、一段上の階層の親」に書く。
複数の場所で並べたいなら、その階層ごとに書く（入れ子になる）。
ある要素が「親の子であり、同時に自分の子の親」になるのは普通のこと。

### 2026/05/22 — Grid のラッパー <div class="layout"> は何のため？

**状況**: 章末演習で <body> の中を <div class="layout"> で囲っていた。
body に直接 display: grid を書けばいいのでは？と思った。

**原因/理解**:
- Grid は「親に display: grid、子に grid-area」の構造なので、並べたい要素を
  まとめる親が必要。
- body でも技術的にはできるが、body にはブラウザのデフォルトスタイルが当たっていて、
  CSS の冒頭でリセット(margin: 0 など)している。そこに Grid の設定まで足すと、
  body の責任が「見た目+タイポグラフィ+レイアウト」と膨らんで管理しにくい。

**解決**: 役割で分ける。
- body → ページ全体の基本設定(色、フォント、行間)
- .layout → レイアウトの組み立て(Grid)
- .header / .main など → 個別パーツの見た目

**教訓**: 「一つの要素に複数の責任を持たせない」。HTML/CSS でも、後の React でも、
同じ考え方が効いてくる。

## つまづき解決ログ

### 2026/05/25 — FlexboxとGridの違いがふんわりとしか理解できていなかった

**状況**: FlexboxとGridの違いが言葉で説明できないレベルでしか分かっていなかった。「行・列」と「縦・横」が同じに聞こえて混乱していた。

**原因**:
- 「行・列」（まとまりの名前）と「縦・横」（方向）が別の切り口の話だと整理できていなかった
- GridをFlexboxの延長として捉えていて、「格子を先に定義する」という発想の違いに気づいていなかった

**解決**:
- 「行・列」と「縦・横」の整理
  - 行（row）= 横方向に伸びるまとまり
  - 列（column）= 縦方向に伸びるまとまり
- FlexboxとGridの発想の違い
  - Flexbox: 要素ありき → 並べ方のルールを決める
  - Grid: 格子ありき → マス目を先に定義してから要素を置く
- Gridはエクセルの表をイメージすると分かりやすい

**教訓**: GridはFlexboxの上位互換ではなく、「先に格子を定義する」という別の発想のツール。どちらを使うか迷ったら「行と列の両方を意識するか？」で判断する。

---

### 2026/05/25 — Flexboxを言葉で説明できるようにしたかった

**状況**: Flexboxがやっていることは理解できるが、言葉で説明できなかった。

**解決**: 「一方向に整列させる。並び方のルールは親が決める」

**教訓**: 自分の言葉で説明できるかどうかが、本当に理解できているかどうかの基準になる。

---

## 定着させたいこと

- **Flexbox = 一方向に整列**。親にルールを書くと、子が従う
- **Grid = 格子を先に定義**してから要素を置く。エクセルの表をイメージ
- **行（row）= 横のまとまり、列（column）= 縦のまとまり**
- Flexbox と Grid は「どちらか」ではなく**組み合わせて使う**のが普通（ページ全体はGrid、ヘッダー内部はFlexなど）
- **`justify-content`** = 並ぶ方向の整列（主軸）
- **`align-items`** = 垂直方向の整列（交差軸）

---

## よく使うパターン

```css
/* 縦横中央寄せ */
.container {
  display: flex;
  justify-content: center;
  align-items: center;
  height: 100vh;
}

/* ナビバー（ロゴ左・メニュー右） */
.navbar {
  display: flex;
  justify-content: space-between;
  align-items: center;
}

/* レスポンシブなカードグリッド（Grid版） */
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 16px;
}

/* ページ全体のレイアウト */
.page {
  display: grid;
  grid-template-areas:
    "header"
    "main"
    "footer";
}
```
